Namespace Engine.Routers

Classes

FloodingPacketRouter

Implements a flooding broadcast routing strategy.

When asked to route a packet the router queries the supplied INetworkTopology for all devices visible to the sender and enqueues a cloned copy addressed to each of them. Flooding uses visibility - not established connections - because it is inherently a broadcast mechanism that needs no prior topology knowledge.

This class is no longer a singleton. The active router instance is stored on Router and is injected into every routing call automatically by Recieve(Packet). To change the routing strategy at runtime replace Router with a different IPacketRouter implementation.

FullMeshNetworkBuilder

A network-formation strategy that connects every pair of mutually visible devices, producing a full-visibility mesh.

This mirrors the original implicit behaviour of the engine where visibility and connectivity were equivalent: if two devices can see each other they are considered connected.

Note: as the number of devices grows, the number of edges grows as O(n?). For large networks consider a limited-degree or spanning-tree builder instead.

NetworkTopology

Default in-memory implementation of IMutableNetworkTopology.

Visibility is computed on-the-fly from device positions and VisibilityDistance; it is never cached so it always reflects the current layout.

Connections are stored as an adjacency set (both directions) and are managed exclusively by the active INetworkBuilder.

SmartFloodingPacketRouter

An improved flooding router that reduces redundant packet copies by applying three optimisations over the naive FloodingPacketRouter:

  1. Direct delivery If the intended destination (To) is directly visible from sender, the packet is sent only to that device - no unnecessary flood copies are made.
  2. No reverse path The packet is never forwarded back to PreviousHop (the device that just sent it here). This eliminates the most common source of duplicate copies in bidirectional flooding.
  3. No source echo The packet is never forwarded back to From (the original source device), preventing useless round-trips.

Together these rules drastically shrink the clone fan-out compared to the naive flood, especially for unicast packets such as ControlPacket, while still guaranteeing delivery in any connected topology with sufficient TTL.

Interfaces

IMutableNetworkTopology

Extends INetworkTopology with mutation methods so that an INetworkBuilder can update the connection graph without exposing mutation to routers or other consumers.

Only INetworkBuilder implementations and SimulationEngine should hold a reference to IMutableNetworkTopology. All other code (routers, statistics, UI) works through the read-only INetworkTopology projection.

INetworkBuilder

Defines the contract for a network-formation (mesh-building) strategy.

A network builder is responsible for deciding which devices establish explicit connections with each other, forming the logical mesh topology. It operates on top of raw visibility data and produces an INetworkTopology that the active IPacketRouter then uses for forwarding decisions.

Examples of algorithms that implement this interface:

  • Full-mesh - every visible pair is connected (current implicit behaviour).
  • Limited-degree mesh - each device connects to at most K neighbours.
  • Minimum spanning tree - minimises total edge weight.
  • Cluster-head / LEACH-style hierarchy.

Like IPacketRouter, the active builder is stored on SimulationEngine and can be swapped at runtime from the UI to compare how different formation strategies affect routing statistics.

INetworkTopology

Represents the logical topology of the network at a given point in time.

The topology separates two concepts that are intentionally kept distinct:

  • Visibility Whether two devices are within radio range of each other. Used for broadcast/discovery and as a precondition for establishing a connection. Determined purely by Euclidean distance and VisibilityDistance.
  • Connection An explicit, mutually agreed logical link between two devices. A connection can only exist between mutually visible devices, but not every visible device is necessarily connected (subject to per-device connection limits, protocol decisions, etc.). Unicast routing and network-topology algorithms operate on connections, not on raw visibility.

Implementations are owned and updated by the active INetworkBuilder and are passed to IPacketRouter on every routing call so that routers never need to touch SimulationEngine directly.

IPacketRouter

Defines the contract for a packet routing strategy.

A router receives a packet from a forwarding device and is responsible for deciding which device(s) the packet should be sent to next. Implementations may use different strategies: flooding broadcast, shortest-path unicast, gradient routing, etc.

The active router is set on SimulationEngine and is consulted by every device whenever it needs to forward a packet. Swapping the router at runtime (e.g. from UI) immediately changes routing behaviour for all subsequent packets, which enables side-by-side protocol comparison on the same network.
